## 2. What's strong

**The architecture earns its complexity.** Each component you built shows up in the results doing real work:

- The hybrid lexical router fires with conf=1.00 on the PowerFlex F004 query (lexical scores 26.3 / 0 / 0) and skips the LLM call entirely — exactly the path the architecture diagram promises.

**The fine-tuning extension is genuinely valuable.** A cross-encoder grader at 7ms beating LLM-as-judge on cost while staying within 1–3 F1 points of it is a clean, reportable result. The ablation framing (opt-in via `USE_LEARNED_GRADER` / `USE_LEARNED_ROUTER`) means you can also tell the reviewer "we kept the LLM path so you can A/B" — that defends against accusations of hiding the comparison.

## 3. Caveats you should disclose (don't get caught on these)

**The 100% router accuracy is partly self-referential.** The router's gold labels are silver labels distilled from Gemini 2.5 Flash itself, then DistilBERT was trained and evaluated against them. So "100%" reads as "DistilBERT learned to imitate Gemini's routing decisions perfectly on a held-out slice." That's still useful — it lets you replace a 1088ms LLM call with a 19ms model — but it is **not** evidence that DistilBERT routes better than humans would. State this explicitly in the report. Suggested phrasing:

> The 100% router accuracy is measured against silver labels generated by Gemini 2.5 Flash. This demonstrates faithful distillation, not ground-truth routing quality. A human-labeled evaluation set would be a natural next step.

**The 84.5% LLM-router baseline is on the same silver-labeled set.** So even the gap (84.5 → 100) is a comparison of "LLM-with-prompt vs DistilBERT-trained-on-LLM-labels." The DistilBERT model reproduces Gemini's labels more reliably than a fresh prompted Gemini call does — which is real but subtle. Don't claim the student beat the teacher on hard reasoning; claim it beat the teacher on label *consistency* and latency.

**The grader gold set size matters.** The grader F1 is more credible because cross-encoder grading against gold relevance labels is closer to the actual task. Make sure you state n in the report — reviewers will ask.

**Class imbalance in the router training set.** industrial=1200, scientific=1200, recipe=847, fallback=800, clarify=800. Macro-F1=1.0 across these classes is striking; mention the per-class confusion matrix or at least the smallest-class recall in the appendix.

## 4. Surprising findings worth investigating

**Synthesis demo looks like a bug.** In the keto query trace, with `enable_synthesis=True`, the supervisor still routed to `clarify` (conf=0.85), and synthesis "didn't actually run because candidate domains were empty." That contradicts the design intent: synthesis is supposed to fan out across domains precisely *when* the supervisor is uncertain. Either:

- the `_route_decision` upgrade from clarify→synthesis isn't firing in this path, or
- `candidate_domains` isn't being populated by the supervisor when confidence is borderline.

Worth tracing before the demo. If you can't fix it in time, **don't show this trace live** — show the recipe/industrial happy paths instead and mention synthesis as "implemented but tuning ongoing."

**CRAG's grade step dominates latency.** In the example trace: retrieve=0.0s, **grade=7.5s**, rewrite=0.0s, generate=3.2s, **validate=5.8s**, total=16.5s. Grading + validation = 13.3s out of 16.5s — about 80% of wall time. This is exactly the cost the learned grader removes (7.5s → milliseconds), which makes the fine-tuning extension feel less like a side-quest and more like the natural P0 optimization. Frame it that way in the talk.

**Basic RAG baseline at 4.0s vs CRAG at 16.5s.** That's a 4× latency cost for the corrective loop. Reviewers will ask whether the extra 12 seconds buys enough. Your answer should be: (a) the hallucination detector caught real errors that basic RAG would have shipped, and (b) with the learned grader the gap closes dramatically. Pre-compute the "CRAG with learned grader" total latency if you have time — that single number is the cleanest defense.

## 5. What to put on which slide

**Slide 13 (Quantitative Results):** Lead with the 2×2 table — Grader (acc, latency) and Router (acc, latency), each vs LLM baseline. The 205× and 100% numbers are the headline. One row, two metrics each, no clutter.

**Slide 14 (Qualitative Results):** The PowerFlex 755T hallucination-catch is the strongest qualitative story — show the answer, highlight the three flagged claims, explain why basic RAG would have shipped them. This is more memorable than another latency chart.

**Slide 15 / Appendix:** Per-stage CRAG timing breakdown (retrieve / grade / rewrite / generate / validate). This is where you tell the story that the learned grader targets the actual bottleneck.

**Don't show:** the synthesis-keto trace (until fixed), the per-class router F1 (unless the smallest class is also 1.0 — verify), and the full 31-disagreement diff between DistilBERT and LLM router (summary stat is enough).
